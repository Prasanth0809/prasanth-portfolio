## Overview

This case study demonstrates how Azure Virtual Networks (VNet) and Network Security Groups (NSG) were configured to isolate and secure cloud infrastructure.

The objective of this implementation was to create a private network environment for Azure resources and enforce strict traffic control using security rules.

This networking setup ensures that cloud resources communicate securely while minimizing exposure to the public internet.

This implementation is part of the **Secure Azure Storage Architecture** project.

---

## Architecture Objective

The networking architecture was designed to achieve the following:

- Isolate Azure resources within a private virtual network  
- Segment network traffic using subnets  
- Control inbound and outbound traffic using Network Security Groups  
- Reduce the public attack surface of the infrastructure  
- Follow Azure networking and security best practices  

---

## Azure Services Used

- Azure Virtual Network (VNet)

# Virtual Network Deployment

## VNet Creation

![Azure Virtual Network Creation](/static/images/azure-vnet-creation-config.png)

A Virtual Network was created to provide a secure and isolated network environment for Azure resources.

The VNet was configured with the following parameters:

- Resource Group: CloudGuard-RG  
- Region: East US  
- Address Space: 10.0.0.0/16  

This network serves as the foundation for hosting and isolating Azure resources within the cloud infrastructure.

---

## Address Space and Subnet Configuration

![Azure VNet Address Space and Subnets](/static/images/azure-vnet-subnet-address-space-config.png)

The address space of the Virtual Network was divided into multiple subnets to logically separate different components of the infrastructure.

Configured subnets include:

- **Default Subnet**
- **Public Subnet**
- **Private Subnet**

Subnet segmentation improves security and allows better control over traffic flow within the network.

---

## Configuration Review Before Deployment

![Azure VNet Review and Create](/static/images/azure-vnet-review-and-create.png)

Before deployment, Azure validates the configuration to ensure that all networking parameters are correctly defined.

The review stage confirms:

- Address space configuration
- Subnet allocation
- Resource group and region selection

This step helps prevent configuration errors before the infrastructure is deployed.

---

## VNet Deployment Success

![Azure VNet Deployment Success](/static/images/azure-vnet-deployment-success.png)

Once validation passed, the Virtual Network was successfully deployed.

The deployed VNet provides a secure internal network layer where Azure resources can communicate privately without relying on public internet exposure.

# Network Security Group Configuration

## Creating Inbound Security Rules

![NSG Inbound Rule Configuration](/static/images/azure-nsg-inbound-rule-config.png)

A Network Security Group (NSG) was configured to control inbound and outbound network traffic.

An inbound rule was created to allow **SSH (Port 22)** access for administrative connectivity while restricting other unauthorized traffic.

Key rule parameters included:

- Protocol: TCP  
- Port: 22  
- Action: Allow  
- Priority: 300  

NSG rules are evaluated based on priority to determine which traffic is allowed or denied.

---

## Verifying NSG Rule Creation

![NSG Rule Added](/static/images/azure-nsg-inbound-rule-added.png)

After creating the rule, it appears in the NSG inbound rule list.

This confirms that the security policy has been successfully applied and will now regulate incoming traffic based on the defined configuration.

---

## Associating NSG with Subnet

![NSG Associated with Subnet](/static/images/azure-subnet-nsg-associated.png)

The Network Security Group was then associated with the **public subnet** inside the Virtual Network.

By attaching the NSG at the subnet level:

- All resources inside the subnet inherit the security rules
- Traffic entering or leaving the subnet is filtered
- Unauthorized access attempts are blocked automatically

This approach ensures centralized traffic control across all resources within the subnet.
